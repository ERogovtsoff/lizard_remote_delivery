# LIZARD — Техническая документация

Описание архитектуры, компонентов и логики приложения. Для разработчиков и тех, кто будет поддерживать проект.

---

## 1. Обзор

LIZARD — Telegram Mini App для магазина одежды с двумя функциями:
1. **Каталог в наличии** — товары для примерки в Беларуси.
2. **Заказ из Китая** — сервис заказа любых вещей через китайские маркетплейсы (POIZON и др.).

Вся коммуникация клиента и менеджера идёт через единого Telegram-бота.

### Стек
- **Фронтенд:** ванильный JavaScript, ES-модули, без сборщика. Хостинг — GitHub Pages.
- **Бэкенд:** Supabase (PostgreSQL + REST API). RLS отключён (прототип).
- **Бот:** Python, aiogram 3.x, обращается к Supabase через REST (httpx).

### Языки и валюты
Интерфейс автоматически переключается RU/EN по языку Telegram. Валюта USD/BYN переключается в настройках, цены хранятся в обеих.

## 4. Логика старта приложения (bootstrap)

При открытии приложения (`app.js`):

1. Инициализируется Telegram WebApp (`tg.ready()`, `tg.expand()`).
2. Показывается **splash-экран** (логотип) — он встроен прямо в HTML с inline-стилями, поэтому виден с первого кадра, до загрузки CSS и JS. Это гарантирует, что новый клиент не увидит «вспышку» главной до решения про онбординг.
3. Определяется, показывать ли онбординг:
   - Если в localStorage есть флаг «онбординг пройден» → сразу главная (БД проверяется в фоне; если суперадмин сбросил флаг — переключаемся на онбординг).
   - Если флага нет → ждём ответ из БД (таймаут 1.5 с). Поле `customers.onboarded`: true → главная, иначе → онбординг.
4. Splash убирается после решения.
5. В фоне грузятся: данные клиента, избранное, корзина, список менеджеров, аватар (предзагрузка в кэш).

**Источник правды онбординга** — `customers.onboarded` в БД. localStorage — быстрый кэш. Суперадмин может сбросить флаг в БД, чтобы показать онбординг заново.

## 5. Модель данных (БД)

### customers — клиенты
- `tg_id` (PK) — Telegram ID
- `first_name`, `last_name`, `username`, `photo_url`
- `purchases_total` — сумма оплаченных заказов в USD
- `purchases_total_byn` — то же в BYN
- `onboarded` — видел ли онбординг
- `preferences` (JSONB) — язык/тема/валюта

### products — товары
- `id` (PK, текст: 'c1', 'p2'...)
- `name_ru/en`, `desc_ru/en`
- `price_usd`, `price_byn`
- `images` (массив URL), `sizes` (массив размеров)
- `stock` (JSONB) — остатки по размерам `{"S":5,"M":0}`
- `is_active` — виден ли клиентам (false = скрытый)
- `badge_text`, `badge_color` — плашка

### orders — заказы
- `id` (PK, автоинкремент)
- `customer_tg_id`
- `total_usd`, `total_byn`, `currency`
- `status` (ENUM, 9 значений — см. раздел 7)
- `is_paid` — оплачен
- `manager_msg_id` — id карточки в чате менеджера
- `created_at`, `updated_at` (последняя активность)

### order_items — позиции заказа
- `order_id`, `product_id`, `size`, `qty`
- `price_usd_snapshot`, `price_byn_snapshot` — цены на момент заказа

### inquiries — обращения (вопросы, не заказы)
- `id` (PK, UUID), `number` (читаемый номер)
- `customer_tg_id`, `type` ('request' | 'product_question'), `product_id`
- `status` ('new' | 'in_progress' | 'closed')
- `manager_msg_id`, `created_at`, `updated_at`

### managers — менеджеры
- `tg_id`, `username`, `is_on_duty`, `chat_id`, `added_by`

### favorites, cart_items — избранное и корзина
Синхронизируются с localStorage, привязаны к `customer_tg_id`.

## 6. Логика приложения по разделам

### Каталог и наличие
Товары грузятся из БД (`products` где `is_active=true`), кэшируются по схеме stale-while-revalidate (`cache/products.js`, TTL 10 с): показываем кэш сразу, обновляем в фоне. Фолбэк — `catalog.json`.

### Размеры и остатки
`products.stock` хранит количество по размерам. На детальной странице размер с остатком 0 — зачёркнут и недоступен. В корзине нельзя набрать больше остатка (подсказка-тост). Если `stock` не задан вовсе — товар считается без ограничений.

### Корзина и оформление
Корзина хранится локально и в `cart_items`. Суммы считаются в обеих валютах. При оформлении создаётся `orders` + `order_items` со снапшотами цен, корзина очищается, клиент переходит в Историю и в чат с ботом (через deep-link `order_<id>`).

### Избранное
Каждая запись — пара (товар, размер). Один товар может быть в избранном с разными размерами.

### История
Объединяет заказы (`orders`) и обращения (`inquiries`), сортирует по дате. Каждый элемент показывает статус. Кэшируется в памяти на сессию (мгновенный показ при повторном открытии, обновление в фоне). Сбрасывается после оформления заказа.

### Профиль и сумма выкупа
Имя/аватар берутся из Telegram (`getUser()`). «Выкуплено на сумму» — из `purchases_total` / `purchases_total_byn` в зависимости от валюты. Сумма обновляется триггером БД при оплате заказа.

### Доступ в админку
`isAdmin()` (tg.js) проверяет: пользователь в `config.ADMIN_USERNAMES` (суперадмин) ИЛИ в таблице `managers` (по tg_id/username). Список менеджеров грузится при старте.

## 7. Статусы

### Заказы (orders.status)
`new` → `in_progress` → `awaiting_payment` → `paid` → `purchasing` → `shipping` → `ready` → `completed` / `cancelled`

Каждый статус имеет текст автоуведомления клиенту. Статус `paid` ставит `is_paid=true` (триггер начисляет сумму выкупа), `cancelled` — снимает.

### Обращения (inquiries.status)
`new` → `in_progress` → `closed`. Уведомления клиенту при «в работе» и «закрыто».

## 8. Бот (bot.py)

### Роли
- **Суперадмин** — задаётся через env `SUPERADMIN_USERNAME`. Полные права.
- **Менеджеры** — таблица `managers`, кешируется в памяти бота.
- `is_manager()` / `is_superadmin()` — проверки прав во всех хендлерах.

### Маршрутизация коммуникации
- Клиент пишет боту → сообщение пересылается всем дежурным менеджерам (заголовок «От @client» + копия). Reply менеджера на заголовок → доставляется клиенту.
- Соответствие «сообщение в чате менеджера ↔ клиент» хранится в `routing.json`.
- Если дежурных нет — fallback на суперадмина.

### Карточки и статусы
Заказы и обращения приходят карточками с inline-кнопками статусов. Нажатие → обновление БД → перерисовка карточки → автоуведомление клиента. Время создания и последней активности показываются в карточке и в `/active`.

### Точки входа (deep-links)
- `/start order_<id>` — клиент оформил заказ
- `/start ask_<product_id>` — вопрос по товару
- `/start request` — запрос на подбор

При каждом создаётся заказ/обращение, рассылается карточка, пишется `manager_msg_id`.

### Команды
См. MANAGER_GUIDE.md, раздел 13.

### Антиспам
Перед созданием обращения проверяется наличие открытого того же типа (`find_open_inquiry`). Один открытый request на клиента, один открытый вопрос на товар. Заказы без ограничений.

### Время и активность
`updated_at` обновляется при: смене статуса, ответе менеджера клиенту, повторном обращении. Время показывается в поясе Беларуси (UTC+3), плюс относительное «X назад».

## 8a. Переменные окружения бота

| Переменная | Назначение |
|-----------|-----------|
| `BOT_TOKEN` | токен бота от @BotFather |
| `WEBAPP_URL` | URL мини-аппа (должен совпадать с доменом в BotFather) |

## 9. Конфигурация (config.js)

- `SUPABASE_URL`, `SUPABASE_ANON_KEY` — подключение к БД.
- `MANAGER_USERNAME`, `ADMIN_USERNAMES` — суперадмин/админы (доступ в админку как фолбэк к таблице managers).
- `BOT_USERNAME` — **обязательно заменить** на реальный юзернейм бота (без @). Используется для deep-links.
- `CATALOG_URL` — путь к сид-каталогу.
- `STORAGE` — ключи localStorage (state, каталог, онбординг).

## 11. Развёртывание

### Приложение
Залить содержимое репозитория на GitHub Pages. При изменении стилей увеличивать `?v=N` у `styles.css` в `index.html` (Telegram агрессивно кэширует).

### База данных
Применить `db/schema.sql` для новой установки, либо миграции по порядку для существующей. Порядок миграций:
1. migration_cart_items
2. migration_onboarded
3. migration_purchases_total_byn

5. migration_drop_requests
6. migration_inquiry_number
7. migration_product_badge
8. migration_product_stock

### Бот
Запустить `bot.py` с переменными окружения на сервере с постоянным аптаймом.

### Telegram
В @BotFather настроить Menu Button с URL мини-аппа и зарегистрировать домен — иначе данные пользователя не передаются (клиент будет «Гостем»).

## 12. Известные ограничения (прототип)

- **RLS отключён** — любой с anon-ключом может читать/писать. Для прода нужно включить RLS + Edge Function для валидации Telegram initData.
- **Ручной заказ** — один товар за раз, количество 1, без выбора размера в боте.
- **Несколько менеджеров** — карточка редактируется у того, кто нажал кнопку; остальные актуализируют через `/active`.
- **Кэш менеджеров в апке** — обновляется при следующем открытии приложения.
